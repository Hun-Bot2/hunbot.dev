# Security Risk Register

Reviewed: 2026-05-12

Do not print or commit secret values. A root `.env` file exists and was not read during this review.

| Risk ID | Severity | Affected files | Scenario | Recommended fix | Test or verification method | Status |
| --- | --- | --- | --- | --- | --- | --- |
| SEC-001 | High | `src/pages/api/views.ts`, `src/components/ViewCounter.astro` | The `slug` query parameter is accepted as a raw Redis key fragment. A client can create arbitrary high-cardinality keys or keys with unexpected separators. | Add a single slug validator that only accepts generated blog content IDs or a strict allowlist pattern. Build Redis keys through a helper that encodes or rejects invalid input. | Unit test allowed and rejected slug cases; smoke test `GET` and `POST` with valid, missing, malformed, and oversized slugs. | Planned |
| SEC-002 | High | `src/pages/api/views.ts` | Duplicate suppression uses client IP plus slug for one hour, but there is no global per-IP rate limit, per-slug rate limit, or bot protection. | Add lightweight rate limiting with short TTL buckets before incrementing view counts. Keep localhost bypass behavior explicit. | API tests for repeated requests, distinct slugs, and rate-limit response shape; monitor Redis key growth in staging. | Planned |
| SEC-003 | Medium | `src/pages/api/views.ts` | IP extraction trusts a request header directly. Depending on proxy behavior, clients may influence the dedupe key. | Prefer the platform-provided client IP when available, parse only the first forwarded IP, and never log derived keys. | Header parsing tests; staging request check through Vercel logs without exposing IP-derived key values. | Planned |
| SEC-004 | Medium | `src/pages/api/views.ts` | Redis credentials missing at runtime still constructs a client and the route fails only on use. | Fail closed with a clear non-secret error path when credentials are absent. Avoid printing values. | Local run without credentials should return a controlled API error and no secret-bearing logs. | Planned |
| SEC-005 | High | `vercel.json`, `src/components/BaseHead.astro`, component inline scripts | CSP allows inline and eval-style scripts. This is currently compatible with the codebase but weakens XSS protection. | Inventory inline scripts, move stable behavior into external scripts or Astro-safe modules, then tighten CSP incrementally. | Build and manual smoke test theme, menu, TOC, analytics, comments, and view counter after each CSP reduction. | Planned |
| SEC-006 | Medium | `src/components/BaseHead.astro`, `src/components/GiscusComments.astro`, `vercel.json` | Third-party scripts for analytics, comments, fonts, and math styles execute or load on public pages. | Keep an explicit third-party inventory, remove unused services, and avoid adding new script domains without CSP and privacy review. | Browser network audit on representative pages; CSP report-only trial before tightening. | Needs manual review |
| SEC-007 | Low | `src/components/GiscusComments.astro`, `src/consts.ts` | Comments depend on public configuration and an external script. Misconfiguration can show comments for the wrong repository/category or fail open visually. | Validate required public comment config at build or render a disabled state. Keep fallback non-sensitive. | Manual page check with comments enabled and disabled; inspect rendered attributes for expected public identifiers only. | Planned |
| SEC-008 | Medium | `src/content/blog/**/*.mdx`, future Library files | MDX content is executable at build time. If AI-generated or third-party content is imported without review, unsafe markup or code can be published. | Treat all generated translations, summaries, and Library content as drafts until human-reviewed. Avoid raw third-party MDX ingestion. | Content validation checklist; code review for any new MDX imports/components. | Needs manual review |
| SEC-009 | Medium | Future Library ingestion docs/data | Library ingestion may mix approved public data with private candidates or raw third-party data. | Store approved public entries separately from private candidate data. Require license, source, attribution, and human review metadata. | Add a validation script before publication once Library schema exists. | Planned |
| SEC-010 | Low | `.gitignore`, root `.env` | Secret files are ignored, but local agents can still read them accidentally. | Keep `.env` ignored, do not inspect it in routine reviews, and use placeholder examples only. | `git status --short` should not show secret files; review docs for absence of secret values. | Implemented |
| SEC-011 | Medium | `src/pages/[lang]/blog/categories/[category].astro` | Category route params are decoded and displayed. Static generation lowers risk, but future dynamic category routes could trust user-supplied path values. | Keep category pages generated from collection data only; encode links and normalize category identifiers if category routes become dynamic. | Build generated category pages; test categories with spaces, case differences, and reserved characters. | Planned |
| SEC-012 | Low | `src/components/BaseHead.astro`, `public/robots.txt` | Analytics and indexing behavior are enabled globally. Privacy and crawler behavior should be deliberate for future Library pages. | Add per-section opt-in or metadata controls before publishing sensitive Library/resource pages. | Manual review of robots, meta robots, and analytics loading on new Library routes. | Planned |
