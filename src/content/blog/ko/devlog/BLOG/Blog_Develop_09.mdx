## Redis 기반 조회수 기능
문제점은 새로고침 시 계속해서 값이 증가한다는 점이다. 이걸 어떻게 해결할 수 있을까?? 내 블로그 구조에 가장 적합한 녀석은 **Redis + TTL**이다.

조회수 중복 방지의 대표적인 방법으로는 Session을 활용하는 방식이 있는데, 내 블로그는 따로 State를 상시 유지하는 백엔드 서버나 DB가 없으며, Vercel의 Edge Serverless 환경을 기반으로 동작한다.

클라이언트의 요청이 들어올 때만 일시적으로 실행되고 바로 없어지는 Stateless 아키텍처이기에, 세션을 관리하려면 별도의 세션 스토리지를 구축하고 매번 쿠키를 파싱하여 서명을 검증하는 오버헤드가 발생한다.

따라서 **IP 및 TTL(Time-To-Live) 기반 락킹(Locking) 기법**을 사용해 구현하려고 한다.

기존의 있던, views.ts파일에 TTL로직만 추가하면 되는 간단한 작업이기에 따로 많은 시간을 들이진 않았다.

### 조회수 처리
그렇다면 기존의 조회수를 계산하는 방식이랑 새로운 조회수를 계산하는 방식이 다른데, 기존에 가지고 있던 조회수는 어떻게 처리하나요?? 라는 생각이 떠올랐고, 이는 `icnr()`로 해결할 수 있다고 한다.

우선, 일반적인 시스템에서 조회수와 같은 기능은 데이터 갱신인데, 데이터 갱신은 
1. DB에서 현재 값을 읽는다.
2. 메모리에서 1을 더한다.
3. 더한 값을 DB에 추가한다.

이런 로직을 거쳐야하는데, Redis의 incr()은 이 과정을 하나의 과정으로 묶어 "Atomic" 하게 처리하기에 내가 원하는 동작을 할 수 있다고 한다.

```typescript
const views = await redis.incr(`pageviews:${slug}`);
```

https://redis.io/docs/latest/commands/set/ 

### 그렇다면 조회수가 없는 글은? 

```typescript
const isNewView = await redis.set(historyKey, '1', { nx: true, ex: 3600 });

   if (!isNewView) {
      const views = await redis.get<number>(`pageviews:${slug}`) || 0;
      return new Response(JSON.stringify({ views, duplicated: true }), { status: 200 });
    }
    const views = await redis.incr(`pageviews:${slug}`);
```

해당 `isNewView` 변수에는 문자열 `"OK"` 또는 `null` 둘 중 하나의 값이 할당된다. 그렇다면, OK라고 온다면 !isNewView가 거짓이기에 바로 view로 넘어가서 조회수를 올리고, 그렇지 않으면 들어가서 숫자를 받아오거나 0으로 설정한다.
