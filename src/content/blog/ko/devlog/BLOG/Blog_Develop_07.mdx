번역기가 필요할까? 구글 번역이 있는데?? 라고 생각했지만, 언어별로 블로그 글이 정리되있는 점이 더 좋지 않을까 해서 영어/일본어 블로그 페이지를 넣게 되었고, 매번 수동으로 번역할 시간이 없고, 어렵기도 해서
성능 좋은 모델을 가져다가 local llm으로 돌려서 비용 부담 없이 번역 기능을 구현해보려고 합니다. 썸네일의 사진은 전체 아키텍처 다이어그램입니다.

흐름은 다음과 같습니다.
1. 내가 vscode로 한국어로 블로그 작성(해당 파일은 로컬 폴더에 저장되어있고, 내가 원할 때 수동으로 commit & push를 함) 
2. terminal에서 명령어로 local llm 실행한 뒤 번역 명령어 입력 (llm 모니터링과 기록이 필수로 남아있어야함) 
3. 번역된 글은 내 vscode내부 /en 폴더와 /jp폴더에 저장됨 
4. AI가 검수하는 동안 나 또한 검수해서 번역 퀄리티 확인 
5. 내가 수동으로 commit & push.

## Local LLM 선택
Ollama의 gamma2 모델을 영어 번역으로, 일본어 번역으로는 Qwen2.5:7b 모델을 선택했습니다. 커뮤니티나 평가 등을 참고해서 가장 성능이 좋다고 평가받는 모델을 선택했습니다.

새벽에 직접 돌려두고 자고 일어나서 확인했는데, 모델들이 말썽을 피우더라고요;; 예를 들어서 "블로그 개발 일지 05"을 일본 버전, 영어 버전에서 LLM이 자기가 하고 싶은 말을 써뒀더라고요;;

우선, 각 모델들이 어떻게 작동하는지 파악해서, 프롬프트 엔지니어링을 하던, 파이썬 코드를 더 꼼꼼하게 나눠서 작성하던 해야 할 것 같습니다.

번역된 글마다 이미지를 또 불러오는 방식이 비효울적일 것 같다라는 생각이 들어 우선, 번역 시스템을 완전하게 구축한 뒤에, 최적화를 생각해봐야 할 것 같습니다.

## Grafana (LLM 모니터링)
![Grafana LLM 모니터링 대시보드](/images/BLOG/grafana01.png)

당연히 llm을 돌리면 얘가 얼마나 빠르게 돌아가는지, 토큰 처리량이 어느 정도인지, GPU 사용률은 어느 정도인지 모니터링이 필요하고, 이를 확인하기 위해서 Grafana를 활용했습니다.
Grafana는 오픈소스 모니터링 툴로, 다양한 데이터 소스와 연동이 가능하고, 시각화가 뛰어나서 많이 사용되고 있고, 처음 사용해봤는데 생각보다 어렵지 않아서 금방 적응할 수 있었습니다.

## 마치며
이번 개발에서는 전체 블로그 아키텍쳐와 로컬 LLM을 활용한 번역 기능을 구현하는 데 중점을 두었습니다. 앞으로 번역 품질을 높이기 위한 프롬프트 엔지니어링과 코드 최적화 작업만 하면 될 것 같습니다.

그리고, 각 언어별 블로그 글마다 이미지를 중복으로 불러오는 부분도 최적화가 필요할 것 같습니다. 다음 개발 일지에서는 **번역 품질 향상과 최적화**에 대해 다뤄보겠습니다. 감사합니다.
